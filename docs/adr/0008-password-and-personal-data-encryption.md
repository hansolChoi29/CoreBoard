# ADR-0008 : 비밀번호와 개인정보 암호화 전략을 분리한 이유

## Date
2026.04.30 (목)


## Context
CoreBoard는 사용자 계정 정보를 관리한다.

사용자 정보에는 비밀번호, 이메일, 전화번호가 포함된다.

처음에는 이 값들을 모두 암호화가 필요한 데이터로 볼 수 있지만, 실제로는 복원 필요 여부가 다르다.

비밀번호는 로그인 시 사용자가 입력한 값과 저장된 값을 비교하면 되므로 원문 복원이 필요하지 않다.

반면 이메일과 전화번호는 서비스 내부에서 사용자 정보 조회, 표시, 안내 등에 사용될 수 있으므로 원문 복원이 필요할 수 있다.

따라서 모든 민감 정보를 같은 방식으로 처리할지, 데이터 성격에 따라 암호화 전략을 분리할지 고민했다.

## Decision
비밀번호는 BCrypt를 사용해 단방향 해시로 저장한다.

이메일과 전화번호는 AES/GCM을 사용해 양방향 암호화로 저장한다.

비밀번호는 원문을 복원하지 않고, 사용자가 입력한 값과 저장된 해시값을 비교하는 방식으로 검증한다.

이메일과 전화번호는 필요한 경우 복호화할 수 있어야 하므로 AES/GCM으로 암호화한다.

AES/GCM을 사용할 때는 같은 평문이 항상 같은 암호문으로 저장되지 않도록 IV 관리가 필요하다.

## Options Considered

### 1. 모든 민감 정보를 BCrypt로 저장
- 장점
    - 단방향이라 원문 유출 위험을 줄일 수 있다.
    - 비밀번호 저장에는 적합하다.

- 단점
    - 이메일과 전화번호를 복원할 수 없다.
    - 서비스 내부에서 원문 조회가 필요한 데이터에는 맞지 않는다.

### 2. 모든 민감 정보를 AES/GCM으로 저장
- 장점
    - 필요한 경우 모든 값을 복호화할 수 있다.
    - 이메일, 전화번호 같은 개인정보 처리에는 사용할 수 있다.

- 단점
    - 비밀번호까지 복호화 가능하게 저장하는 것은 적절하지 않다.
    - 키 관리가 잘못되면 비밀번호 원문 노출 위험이 커진다.
    - 비밀번호는 애초에 복원이 필요 없는 데이터다.

### 3. 데이터 성격에 따라 BCrypt와 AES/GCM 분리 (선택)
- 장점
    - 비밀번호는 복원 불가능한 방식으로 안전하게 저장할 수 있다.
    - 이메일과 전화번호는 필요한 경우 복호화할 수 있다.
    - 데이터의 사용 목적에 맞게 보안 전략을 다르게 적용할 수 있다.

- 단점
    - 암호화 방식이 둘로 나뉘어 관리 포인트가 늘어난다.
    - AES/GCM 키와 IV 처리 기준을 별도로 관리해야 한다.
    - 테스트에서도 해시 검증과 복호화 검증을 각각 다뤄야 한다.

## Consequences
- 좋은 점
    - 비밀번호와 개인정보의 보안 요구사항을 다르게 반영할 수 있었다.
    - 비밀번호 원문 복원 가능성을 제거했다.
    - 이메일과 전화번호는 필요한 경우 복호화할 수 있게 했다.
    - 같은 "암호화"라도 데이터 성격에 따라 전략이 달라진다는 기준을 설계에 반영했다.

- 나쁜 점 (트레이드오프)
    - BCrypt와 AES/GCM을 각각 테스트해야 한다.
    - AES/GCM 키 관리가 필요하다.
    - 암호화 유틸의 책임이 커질 수 있다.

## Evidence (선택)
- 비밀번호: BCrypt 해시값 저장
- 이메일 / 전화번호: AES/GCM 암호화 저장
- BCrypt cost=12 적용
- AES/GCM에서 IV를 매번 새로 생성해 같은 평문도 다른 암호문으로 저장
