# ADR-0006 : 인증 흐름 처리를 AOP가 아닌 HandlerInterceptor에서 수행한 이유

## Date
2026.04.30 (목)

## Status
Accepted

## Context
CoreBoard는 Spring Security를 바로 적용하지 않고, 인증/인가 흐름을 직접 구현하는 방식을 선택했다. 이 과정에서 JWT 검증과 관리자 권한 확인 로직을 어디에 위치시킬지 결정이 필요했다.
후보는 AOP와 HandlerInterceptor 두 가지였다.
핵심적인 차이는 언제, 어디서 개입하느냐다.

- HandlerInterceptor는 HTTP 요청이 컨트롤러에 도달하기 전/후 시점에 개입한다. 요청 헤더, URL, HTTP Method 같은 웹 요청 정보에 자연스럽게 접근할 수 있다.
- AOP는 Spring Bean의 메서드 실행 시점을 기준으로 공통 로직을 끼워 넣는다. 로깅, 트랜잭션, 실행 시간 측정 같은 용도에 어울린다.

CoreBoard에서 인증이 필요한 이유를 생각해보면, "특정 서비스 메서드를 실행하기 전에 뭔가 하고 싶다"가 아니라, "유효하지 않은 요청 자체를 컨트롤러에 들어오기 전에 막고 싶다" 는 것이다.
이는 AOP보다 HTTP 요청 처리 파이프라인에서의 전처리 문제에 더 가깝다고 판단했다.

## Decision
JWT 인증 흐름은 AOP가 아닌 HandlerInterceptor에서 처리한다.
구체적으로는 다음 두 가지를 Interceptor에서 담당한다.

1. 컨트롤러 진입 전, 요청 헤더의 JWT 토큰을 검증한다.
2. 검증된 사용자 정보를 이후 요청 처리 흐름에 전달한다.

관리자 API(/admin/**)는 요청 경로와 사용자 권한 정보를 기준으로 접근 가능 여부를 판단한다.
이 결정의 핵심 근거는 한 문장으로 정리하면 이렇다.

> 인증/인가는 "메서드 실행 전후에 붙는 부가기능"이 아니라, "HTTP 요청이 컨트롤러에 도달하기 전에 처리해야 하는 요청 전처리 책임" 이다.

## Options Considered

### 1. AOP로 인증/인가 처리
- 장점
    - 여러 컨트롤러나 서비스 메서드에 공통 검증 로직을 쉽게 적용할 수 있다.
    - 어노테이션 기반으로 특정 메서드에만 권한 검증을 선택적으로 붙일 수 있다.
    - 로깅, 트랜잭션처럼 공통 관심사를 분리하는 용도에 자연스럽다.

- 단점
    - 인증의 기준이 되는 것은 요청 헤더, URL, HTTP Method인데, AOP는 이 정보에 접근하는 구조가 어색하다.
    - 메서드 실행 시점에 가까운 도구이기 때문에, 컨트롤러 진입 전에 요청을 차단한다는 흐름이 `HandlerInterceptor`보다 자연스럽지 않다.
    - 검증된 사용자 정보를 이후 웹 요청 처리 흐름에 전달하는 방식도 명확하지 않다.

### 2. HandlerInterceptor로 인증/인가 처리 (선택)
- 장점
    - 컨트롤러 실행 전 시점에 요청을 가로채 인증 여부를 확인할 수 있다.
    - `HttpServletRequest`를 통해 요청 헤더, URI, HTTP Method에 자연스럽게 접근할 수 있다.
    - 인증 실패 시 컨트롤러에 진입하기 전에 요청을 차단할 수 있어, 불필요한 서비스 레이어 실행을 막는다.
    - `request.setAttribute()`를 통해 검증된 사용자 정보를 이후 처리 흐름에 쉽게 전달할 수 있다.
    - `/admin/**` 같은 URL 패턴 기반 권한 검증을 적용하기에 적합하다.

- 단점
    - 메서드 단위의 세밀한 권한 제어에는 AOP나 Spring Security보다 불리할 수 있다.
    - 권한 정책이 복잡해지면 Interceptor 내부 조건이 많아질 수 있다.
    - Spring Security가 제공하는 인증 객체, SecurityContext, 세부 권한 관리 기능은 직접 제공하지 못한다.

## Consequences
- 좋은 점
    - HTTP 요청이 컨트롤러에 도달하기 전에 인증 실패 요청을 차단할 수 있었다.
    - 요청 헤더 확인, 토큰 검증, 사용자 정보 전달 흐름을 명확히 설계할 수 있었다.
    - Interceptor와 AOP의 적용 지점 차이를 기준으로 인증 흐름 위치를 선택했다는 점을 설명할 수 있다.
    - 관리자 API에 대해 URL 패턴 기반 권한 검증을 적용할 수 있었다.

- 나쁜 점 (트레이드오프)
    - 메서드별 세밀한 권한 제어가 필요해지면 구조를 재검토해야 한다.
    - 권한 정책이 많아질수록 Interceptor가 비대해질 수 있다.
    - 운영 수준의 보안 요구사항이 커지면 Spring Security 도입이 필요할 수 있다.

## Evidence (선택)
- 인증 처리를 AOP가 아닌 `HandlerInterceptor` 기반 요청 전처리 흐름으로 분리
- HTTP 요청 헤더와 URL 패턴을 기준으로 인증/인가 흐름 구성
- 관리자 API에 대해 URL 패턴 기반 권한 검증 적용
