# ADR-0005 : Admin 초기 생성 및 권한 전환 정책

## Date

2026.04.29 (수)

## Context

- 권한 책임 분리: 게시판 관리 등 운영 기능을 수행할 관리자 계정 정의가 필요하다.
- 가입 흐름 혼재: 일반 회원가입(USER) 로직에 관리자 생성을 포함할 경우, 운영 계정 생성 기준이 모호해지고 보안 취약점이 발생할 수 있다.
- 운영 가용성 리스크: 마지막 관리자의 권한을 회수할 경우, 시스템 관리 주체가 사라지는 관리자 부재상태가 우려된다.

## Decision

- 엔드포인트 격리: 일반 가입은 USER 생성으로 한정하고, 관리자 생성은 /admin/setup으로 완전 분리한다.
- Bootstrapping 제약: /admin/setup은 최초 1회(ADMIN 0명일 때)만 허용하며, 이미 관리자가 존재할 경우 요청을 거부한다.
- 권한 승격 구조: 추가 관리자는 /admin/setup이 아닌, 기존 ADMIN이 일반 유저를 승격시키는 '권한 전환 API'를 통해서만 생성한다.
- 최소 인원 보존: ADMIN 계정이 1명뿐인 경우, 해당 계정의 권한 회수(ADMIN → USER)를 차단한다.
- 예외 규격화: 운영 효율을 위해 에러 코드와 사용자 메시지를 분리하여 응답한다.

## Options Considered

### 1. 일반 회원가입에서 ADMIN role을 생성하도록 처리

- 장점
    - 별도 관리자 생성 API가 필요하지 않다.

- 단점
    - USER 가입 흐름과 ADMIN 생성 흐름이 섞인다.
    - 일반 가입 흐름에서 ADMIN 계정이 생성될 위험이 있다.
    - 운영 관리 계정 생성 기준이 코드 흐름상 명확하지 않다.

### 2. `/admin/setup`으로 관리자 생성 흐름 분리

- 장점
    - USER 가입 흐름과 ADMIN 생성 흐름을 분리할 수 있다.
    - Board 생성 및 관리 기능을 수행할 관리자 계정 기준을 명확히 둘 수 있다.
    - 최초 관리자 생성 이후에는 기존 ADMIN의 권한 전환 API로 추가 관리자를 만들 수 있다.

- 단점
    - `/admin/setup`을 제한하지 않으면 ADMIN 계정이 이미 존재해도 추가 생성이 가능하다.
    - 따라서 ADMIN 존재 여부를 기준으로 추가 setup 요청을 막는 정책이 필요하다.

### 3. DB에 직접 ADMIN 계정 insert

- 장점
    - 별도 API를 만들지 않아도 된다.
    - 초기 관리자 계정을 빠르게 만들 수 있다.

- 단점
    - 애플리케이션 API 흐름 밖에서 데이터가 생성된다.
    - 운영 환경에서 재현 가능성과 관리 기준이 떨어진다.
    - 계정 생성 시 암호화, 비밀번호 인코딩 등 애플리케이션 로직을 우회할 수 있다.

## Consequences

- 좋은 점
    - 명확한 주체: 운영 기능 수행 주체를 확실히 분리하여 보안성 강화.
    - 안정적 운영: 시스템 내 관리자 0명 상태를 방지하여 영속성 확보.
    - 확장성: setup(초기화)과 전환(운영) 로직을 구분하여 향후 복잡한 권한 정책 도입 시 유리함.

- 나쁜 점 (트레이드오프)
    - 오버헤드: setup API 호출 시마다 관리자 존재 여부 확인을 위한 DB 조회가 발생함.
    - 미구현 사항: 권한 변경 이력(Audit Log) 및 다중 관리자 환경에서의 자기 자신 권한 회수 제한은 향후 과제로 남음.

## Evidence

- 초기 생성 제한: 관리자 존재 시 ADMIN_ALREADY_EXISTS 응답 확인.
- 권한 회수 제한: 마지막 관리자 강등 시도 시 409 Conflict 응답 확인.

```json
{
  "success": false,
  "message": "LAST_ADMIN_CANNOT_BE_DEMOTED",
  "data": {
    "code": "409",
    "errors": [
      {
        "field": "LAST_ADMIN_CANNOT_BE_DEMOTED",
        "reason": "마지막 관리자 계정을 일반 사용자로 변경할 수 없습니다."
      }
    ]
  }
