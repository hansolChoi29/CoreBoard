# ADR-0002 : 게시글 목록 조회에서 조회 목적별 페이지네이션 전략을 분리한 이유


## Date
2026.04.24 (금)

## Context
전체 게시글 조회에 Keyset pagination을 적용했다.
하지만 결정 당시, 사용자의 조회 방식이나 게시판별 UX를 고려했다기 보단, Offset보다 Keyset이 빠르다는 이유에 가까웠다.

Board의 유형별로 나누어 사용자의 편의를 신경쓰기로 했다.

자유게시판과 Q&A는 사용자가 목록을 탐색하거나 검색 결과에서 특정 페이지로 이동할 가능성이 높다.

따라서 모든 목록 조회에 하나의 페이지네이션 전략을 적용하는 것은 적절하지 않다고 판단했다.

다만 깊은 페이지로 갈수록 Offset 비용이 커질 수 있으므로 정렬 기준에 인덱스를 적용하고, 너무 깊은 페이지 접근은 검색/필터로 유도한다.

댓글은 전체 페이지 수보다 다음 댓글 존재 여부가 중요하므로 Slice를 사용한다.

최신글 더보기나 무한스크롤 API가 필요해지면 Keyset 방식을 별도 API로 제공한다.

## Decision
조회 대상과 사용 흐름에 따라 Page, Slice, Keyset을 분리해서 사용한다.

자유게시판 전체조회: Offset + Page

Q&A 전체조회: Offset + Page

공지사항 전체조회: Offset + Page

댓글: Slice

최신글 더보기 API: Keyset (추후)

## Options Considered
### 1. 기존 Keyset pagination 유지
- 장점: 기존 구현을 유지할 수 있고, 뒤쪽 데이터 조회에서도 성능이 안정적임
- 단점: 페이지 번호 이동이 어려워 자유게시판, Q&A, 공지사항 같은 웹 게시판 목록 UX에 맞지 않음

### 2. 게시글 목록을 Offset + Page로 전환
- 장점: 페이지 번호 이동, 전체 개수 표시, 검색 결과 탐색이 가능해 웹 게시판 UX에 적합함
- 단점: 데이터가 많아질수록 깊은 페이지 조회 비용이 증가할 수 있음

### 3. 조회 목적에 따라 Page, Slice, Keyset 분리 (선택)
- 장점: 자유게시판/Q&A/공지사항은 페이지 번호 기반 조회를 제공하고, 댓글은 더보기 흐름에 맞게 조회할 수 있음
- 단점: 조회 방식이 나뉘므로 API 응답 형식과 테스트 케이스가 증가함

## Consequences
- 좋은 점: 게시판별 사용 방식에 맞는 페이지네이션 전략을 적용할 수 있음
- 나쁜 점 (트레이드오프): Offset 방식과 Keyset 방식이 함께 존재하므로 API 설계와 테스트 케이스가 증가함
