# ADR-0010 : 테스트 신뢰도를 위해 H2 대신 Testcontainers MySQL을 선택한 이유

## Date
2026.04.30 (목)

## Status
Accepted

## Context
CoreBoard는 MySQL을 실제 데이터베이스로 사용한다.

테스트 환경에서는 빠른 실행을 위해 H2 인메모리 DB를 사용할 수도 있다.

H2는 인메모리 DB로 빠르게 테스트를 실행할 수 있고, MySQL compatibility mode도 제공한다.

하지만 compatibility mode를 사용하더라도 H2와 MySQL은 서로 다른 데이터베이스다.

SQL 방언, 제약조건 처리, 타입 처리, 인덱스 동작, 트랜잭션 처리 방식에서 차이가 발생할 수 있다.

이 경우 테스트는 통과하지만 실제 MySQL 환경에서는 실패하는 문제가 생길 수 있다.

CoreBoard는 인증, 게시글, 게시판, 트랜잭션, 예외 흐름을 실제 DB 조건에 가깝게 검증하는 것이 중요했다.

따라서 테스트 속도를 우선해 H2를 사용할지, 운영 환경과 같은 MySQL 계열 데이터베이스를 Testcontainers로 띄워 검증할지 결정해야 했다.

## Decision
통합 테스트에서는 H2 대신 Testcontainers 기반 MySQL을 사용하기로 했다.

테스트 실행 시 MySQL 컨테이너를 띄우고, 운영 환경과 같은 MySQL 계열 데이터베이스에서 쿼리, 제약조건, 트랜잭션 흐름을 검증한다.

이 결정의 핵심 이유는 테스트 실행 속도보다 실제 DB 환경과의 일치성과 통합 테스트 신뢰도를 더 중요하게 봤기 때문이다.

Testcontainers는 Docker 컨테이너를 통해 테스트에 필요한 실제 의존성을 일회성 환경으로 구성할 수 있다.

따라서 로컬 개발자 DB 상태에 의존하지 않고, 테스트마다 재현 가능한 MySQL 기반 통합 테스트 환경을 구성할 수 있다.

## Options Considered

### 1. H2 인메모리 DB 사용
- 장점
  - 테스트 실행 속도가 빠르다.
  - 별도 Docker 환경이 없어도 테스트하기 쉽다.
  - 설정이 비교적 간단하다.
  - 단위 테스트나 단순 Repository 테스트에는 부담이 적다.

- 단점
  - MySQL과 SQL 방언이 다르다.
  - MySQL compatibility mode를 사용하더라도 실제 MySQL과 완전히 동일하지 않다.
  - 실제 DB에서만 발생하는 제약조건, 타입, 쿼리 문제를 놓칠 수 있다.
  - 테스트 통과가 실제 운영 환경 동작을 보장하지 못할 수 있다.

### 2. 로컬 MySQL 직접 사용
- 장점
  - 실제 MySQL에서 테스트할 수 있다.
  - Testcontainers보다 컨테이너 기동 오버헤드가 적을 수 있다.
  - 로컬에서 직접 DB 상태를 확인하기 쉽다.

- 단점
  - 개발자 로컬 환경에 의존한다.
  - 테스트 실행 전 DB 상태를 매번 맞춰야 한다.
  - 테스트 간 데이터 오염이 발생할 수 있다.
  - CI 환경에서 동일한 조건을 재현하기 어렵다.

### 3. Testcontainers 기반 MySQL 사용 (선택)
- 장점
  - 실제 MySQL과 같은 계열의 DB 환경에서 테스트할 수 있다.
  - 테스트마다 격리된 DB 환경을 구성할 수 있다.
  - CI에서도 동일한 방식으로 테스트 환경을 만들 수 있다.
  - SQL 방언 차이로 인한 테스트 신뢰도 문제를 줄일 수 있다.
  - 로컬 DB 상태에 의존하지 않아 재현성이 높다.

- 단점
  - Docker 실행이 필요하다.
  - 컨테이너 기동 시간이 추가된다.
  - H2보다 테스트 실행 속도가 느릴 수 있다.
  - 테스트 환경 설정과 프로파일 관리가 필요하다.

## Consequences
- 좋은 점
  - MySQL 기준의 실제 쿼리 동작을 검증할 수 있다.
  - H2와 MySQL의 방언 차이로 인한 테스트 오판을 줄일 수 있다.
  - 통합 테스트 신뢰도가 높아졌다.
  - 인증/게시글/게시판 흐름을 실제 DB 조건에 가깝게 검증할 수 있다.
  - 로컬 DB 상태와 무관하게 재현 가능한 테스트 환경을 구성할 수 있다.

- 나쁜 점 (트레이드오프)
  - Docker가 실행 중이어야 한다.
  - 테스트 실행 시간이 증가할 수 있다.
  - 컨테이너 설정과 테스트 프로파일 관리가 필요하다.
  - 단순 단위 테스트까지 Testcontainers에 의존하면 테스트 피드백 속도가 느려질 수 있다.

## Evidence (선택)
- Testcontainers MySQL 기반 통합 테스트 구성
- 컨테이너 재사용 설정으로 테스트 실행 시간 부담 완화
- 테스트 프로파일에서 BCrypt cost를 낮춰 인증 흐름 테스트 시간 단축
- 도메인 레이어 테스트 커버리지 100% 달성
