# ADR-0001 : 운영 중 변경 가능한 게시판 메뉴를 DB 데이터로 관리한 이유

## Date
2026.04.24 (금)

## Context
기존 CoreBoard는 게시판 메뉴와 게시글이 명확히 분리되지 않은 단일 Board 구조였다.

피드백을 반영하면서 공지사항, 자유게시판, Q&A처럼 여러 게시판 메뉴를 운영할 수 있는 구조가 필요하다고 판단했다.

처음에는 게시판 메뉴를 `BoardType` enum으로 관리하는 방식을 고려했다.

하지만 enum으로 관리하면 새로운 게시판 메뉴가 추가될 때마다 코드 수정, 테스트, 빌드, 재배포가 필요하다.

반면 게시판 메뉴는 서비스 운영 중 추가되거나 비활성화될 수 있는 운영 데이터에 가깝다.

따라서 게시판 메뉴를 코드에 고정할 값으로 볼 것인지, DB 데이터로 관리할 값으로 볼 것인지 고민했다.

## Decision
게시판 메뉴는 enum이 아니라 Board 엔티티로 DB에서 관리하기로 했다.

Board는 공지사항, 자유게시판, Q&A 같은 게시판 메뉴를 의미한다.

Post는 특정 Board에 작성되는 게시글을 의미한다.

이를 통해 같은 게시글 구조를 사용하는 게시판 메뉴는 서버 재배포 없이 DB 데이터 추가로 확장할 수 있게 한다.


## Options Considered
### 1. Enum으로 게시판 메뉴 관리
- 장점: 구현이 단순하고 코드에서 허용된 값만 사용할 수 있어 타입 안정성이 높음
- 단점: 새로운 게시판 메뉴가 추가될 때마다 코드 수정, 테스트, 빌드, 재배포가 필요함

### 2. Board 엔티티로 DB 관리 (선택)
- 장점: 같은 게시글 구조를 사용하는 게시판 메뉴는 운영 중 DB 데이터 추가로 확장 가능함
- 단점: 게시판 이름/slug 중복 검증, 비활성화 정책, 권한 정책 등 관리 포인트가 증가함


## Consequences
- 좋은 점: 단순 게시판 메뉴 추가와 새로운 기능 개발을 분리할 수 있음
- 나쁜 점 (트레이드오프): enum 방식보다 구조가 복잡해지고 게시판 데이터 정합성을 관리해야 함
