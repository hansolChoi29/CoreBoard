# ADR-0001 : 운영 중 변경 가능한 게시판 메뉴를 DB 데이터로 관리한 이유

## Date
2026.04.24 (금)

## Context
기존 CoreBoard는 게시판 메뉴와 게시글이 명확히 분리되지 않은 단일 Board 구조였다.

피드백을 반영하면서 공지사항, 자유게시판, Q&A, 자료실, 갤러리처럼 여러 게시판 메뉴를 운영할 수 있는 구조가 필요하다고 판단했다.

처음에는 공지사항, 자유게시판, Q&A 같은 게시판 메뉴 자체를 enum으로 관리하는 방식을 고려했다.

하지만 게시판 메뉴를 enum으로 관리하면 새로운 메뉴가 추가될 때마다 코드 수정, 테스트, 빌드, 재배포가 필요하다.

또한 갤러리나 자료실처럼 게시판마다 입력 규칙이 달라질 수 있는데, 이를 게시판 유형 enum으로 계속 추가하면 새로운 유형이 생길 때마다 다시 코드에 의존하게 된다.

반면 게시판 메뉴와 기본 운영 정책은 서비스 운영 중 추가되거나 변경될 수 있는 운영 데이터에 가깝다.

따라서 게시판 메뉴와 게시판별 기본 정책을 코드에 고정할 값으로 볼 것인지, DB 데이터로 관리할 값으로 볼 것인지 고민했다.

## Decision
공지사항, 자유게시판, Q&A 같은 게시판 메뉴 정보는 enum으로 코드에 고정하지 않고, `Board` 테이블의 데이터로 관리하기로 했다.

`Board`는 공지사항, 자유게시판, Q&A, 자료실, 갤러리 같은 게시판 메뉴를 의미한다.

`Post`는 특정 `Board`에 작성되는 게시글을 의미한다.

게시판마다 작성 권한, 본문 길이, 첨부파일 필요 여부, 허용 파일 타입, 최대 첨부파일 개수 같은 기본 정책이 달라질 수 있으므로, 이를 enum으로 고정하지 않고 `Board`의 설정값으로 관리한다.

이를 통해 같은 `Post`와 `Attachment` 구조로 표현 가능한 게시판 메뉴는 서버 재배포 없이 DB 데이터 추가만으로 확장할 수 있게 한다.

예를 들어 갤러리 게시판은 별도의 `gallery_post` 테이블을 만들지 않고, `Board` 설정에서 첨부파일 필수 여부와 허용 파일 타입을 이미지로 제한해 처리한다.

다만 지도 게시판처럼 위도, 경도, 마커, 거리 계산 등 완전히 다른 데이터 구조가 필요한 기능은 공통 게시판 설정으로 해결하지 않고 별도 기능 개발 대상으로 분리한다.

## Options Considered

### 1. Enum으로 게시판 메뉴 관리
- 장점: 구현이 단순하고 코드에서 허용된 값만 사용할 수 있어 타입 안정성이 높음
- 단점: 새로운 게시판 메뉴가 추가될 때마다 코드 수정, 테스트, 빌드, 재배포가 필요함

### 2. 게시판 유형 enum으로 동작 분기
- 장점: 공지사항, Q&A, 갤러리처럼 동작이 다른 게시판을 코드에서 명확하게 구분할 수 있음
- 단점: 새로운 게시판 유형이 생길 때마다 enum 추가, 검증 로직 추가, 테스트, 빌드, 재배포가 필요함

### 3. Board 엔티티 + 설정값 기반 관리 (선택)
- 장점: 게시판 메뉴와 기본 정책을 DB 데이터로 관리할 수 있어, 공통 Post/Attachment 구조로 표현 가능한 게시판은 운영 중 추가 가능함
- 장점: 갤러리, 자료실처럼 첨부파일 정책만 다른 게시판은 별도 테이블 없이 설정값으로 처리할 수 있음
- 단점: enum 방식보다 구조가 복잡해지고, 설정값 검증과 데이터 정합성 관리가 필요함

## Consequences
- 좋은 점: 단순 게시판 메뉴 추가와 새로운 기능 개발을 분리할 수 있음
- 좋은 점: `이벤트게시판`, `자료실`, `갤러리`처럼 공통 게시글/첨부파일 구조를 쓰는 메뉴는 DB 데이터 추가만으로 확장 가능함
- 좋은 점: 게시판마다 작성 권한, 본문 길이, 첨부파일 정책을 다르게 설정할 수 있음
- 좋은 점: 게시판 추가를 위해 매번 새 테이블을 만들거나 서버를 재배포하지 않아도 됨
- 나쁜 점 (트레이드오프): enum 방식보다 구조가 복잡해지고 게시판 설정 데이터의 정합성을 관리해야 함
- 나쁜 점 (트레이드오프): 모든 게시판 유형을 무배포로 해결할 수는 없으며, 완전히 다른 데이터 구조가 필요한 기능은 별도 개발이 필요함
