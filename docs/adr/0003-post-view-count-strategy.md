# ADR-0003 :  조회수 증가로 인해 읽기 API에 쓰기 부하가 생기는 문제와 1차 구현 기준

## Date

2026.4.28 (화)

## Context

게시글 조회수는 단순한 숫자처럼 보이지만, 실제로는 읽기 요청에 쓰기 작업을 추가하는 기능이다.

상세조회 API가 호출될 때마다 `post.view_count`를 증가시키면 구현은 단순하지만, 조회 요청이 많아질수록 DB update 부하가 증가한다.

또한 같은 사용자의 새로고침, 비로그인 사용자의 반복 접근, 작성자 본인 조회, 봇 접근 등을 모두 조회수로 볼 것인지 기준이 필요하다.

따라서 조회수를 정확한 열람 기록으로 관리할지, 가벼운 인기 지표로 관리할지 결정해야 했다.

## Decision

1차 구현에서는 `Post` 테이블에 `viewCount` 컬럼을 두고, 게시글 상세조회 API가 성공했을 때 바로 증가시키는 방식을 선택했다.

전체 게시글 목록 조회는 사용자가 게시글을 실제로 읽은 것이 아니라 목록에 노출된 상태에 가깝다고 판단했다. 따라서 전체조회에서는 조회수를 증가시키지 않고, 단건 상세조회에서만 조회수를 증가시킨다.

조회수는 정확한 열람 로그가 아니라 가벼운 인기 지표로 정의한다. 그래서 1차 구현에서는 동일 사용자 또는 동일 IP의 반복 조회를 중복 제거하지 않는다.

Redis TTL이나 ViewLog 테이블 방식은 중복 조회 방지와 통계 분석에는 유리하지만, 현재 CoreBoard 규모에서는 운영 복잡도가 크다고 판단해 보류했다.

## Options Considered

### 1. Post 테이블에 viewCount 컬럼을 두고 상세조회 시 바로 증가

- 장점
    - 구현이 단순하고 게시글 조회 응답에 뷰카운트를 바로 포함할 수 있다
    - 별도 테이블이나 redis가 필요없다
    - 현재 CoreBoard 규모에 적합하다

- 단점
    - 상세조회마다 업데이트 쿼리가 발생한다
    - 읽기 트래픽이 쓰기 부하로 바뀐다
    - 동일 사용자의 새로고침도 모두 카운트 된다

### 2. ViewLog 테이블을 따로 둔다

|post_view_log|
|---|
|id|
|post_id|
| user_id|
| ip|
|viewed_at|

- 장점
    - 누가 언제 봤는지 기록할 수 있다
    - 중복 조회 기준을 나중에 정교하게 만들 수 있다
    - 통계 분석에 유리하다

- 단점
    - 조회마다 로그 row가 쌓인다
    - 테이블 크기가 빠르게 커진다
    - 단순 조회수 기능 치고 구조가 무거워진다
    - 로그 저장 실패와 게시글 조회 성공을 어떻게 볼지도 정해야 한다

### 3. Redis로 중복 조회를 막고 조회수는 나중에 DB 반영

view:post:1:user:10 → TTL 30분

- 장점
    - 동일 사용자 반복 조회를 일정 시간 동안 막을 수 있다
    - DB update 부하를 줄일 수 있다
    - 인기글처럼 조회수가 자주 바뀌는 값에 유리하다

- 단점
    - Redis 장애 시 조회수 처리 정책이 필요하다
    - Redis 값과 DB 값 사이 정합성 문제가 생긴다
    - 주기적으로 DB에 반영하는 배치가 필요할 수 있다
    - 현재 프로젝트 규모에서는 운영 복잡도가 커진다

## Consequences

- 좋은 점
  - 구현이 단순하다.
  - 게시글 상세조회 응답에 조회수를 바로 포함할 수 있다.
  - 별도 ViewLog 테이블이나 Redis 없이 조회수 기능을 제공할 수 있다.
  - 현재 CoreBoard 규모에서는 기능 대비 복잡도가 적절하다.

- 나쁜 점 (트레이드오프)
  - 게시글 상세조회마다 `UPDATE` 쿼리가 발생한다.
  - 읽기 API처럼 보이는 상세조회 API가 실제로는 쓰기 부하를 만든다.
  - 동일 사용자의 새로고침도 조회수로 카운트된다.
  - 동시 조회가 많아질 경우 조회수 증가 경쟁이나 DB write 부하를 추가로 검토해야 한다.
  - 정확한 사용자 행동 로그나 통계 분석에는 적합하지 않다.
