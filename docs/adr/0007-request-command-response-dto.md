# ADR-0007 : 읽기/쓰기 흐름의 책임 분리를 위해 DTO를 request-command-response로 나눈 이유

## Date
2026.04.30 (목)

## Context
초기 구현에서는 외부 요청 DTO를 서비스 계층까지 그대로 전달하거나, 입력 DTO와 응답 DTO의 역할이 명확히 구분되지 않는 문제가 있었다.

하지만 상태를 변경하는 요청과 데이터를 조회해 반환하는 응답은 목적이 다르다.

쓰기 흐름은 사용자의 입력값을 받아 게시글 생성, 수정, 권한 변경처럼 시스템 상태를 변경하는 데 집중한다.

반면 읽기 흐름은 클라이언트 화면에 필요한 데이터를 안정적인 응답 형태로 반환하는 데 집중한다.

이 둘을 같은 DTO나 엔티티 중심으로 처리하면, 서비스 계층이 HTTP 요청 형식에 의존하거나 내부 엔티티 구조가 외부 응답에 노출될 수 있다.

또한 외부 API 스펙이 바뀔 때 서비스 입력 모델까지 함께 흔들리고, 반대로 엔티티 필드가 바뀔 때 응답 구조까지 영향을 받을 수 있다.

따라서 외부 요청 입력, 서비스 내부 입력, 외부 응답 출력을 분리해 읽기/쓰기 흐름의 책임을 명확히 할 필요가 있었다.

## Decision
DTO를 `request → command → response` 구조로 분리하기로 했다.

- `request`: 클라이언트의 HTTP 요청을 받는 외부 입력 DTO
- `command`: 상태 변경 또는 비즈니스 처리를 위해 서비스 계층에 전달하는 내부 입력 모델
- `response`: 클라이언트에게 반환하는 외부 출력 DTO

컨트롤러는 request DTO를 받은 뒤 필요한 검증과 변환을 거쳐 command DTO를 서비스에 전달한다.

서비스는 HTTP 요청 형식에 직접 의존하지 않고, command를 기준으로 비즈니스 흐름을 처리한다.

조회 결과는 response DTO로 분리해 클라이언트에게 필요한 값만 반환한다.

이 구조는 완전한 CQRS 적용이 아니라, 읽기와 쓰기의 데이터 목적이 다르다는 점을 반영한 DTO 수준의 책임 분리다.

## Options Considered

### 1. Request DTO를 서비스까지 그대로 전달
- 장점
  - 구현이 단순하다.
  - DTO 클래스 수가 적다.
  - 작은 CRUD에서는 빠르게 개발할 수 있다.

- 단점
  - 서비스 계층이 HTTP 요청 형식에 의존하게 된다.
  - API 입력 형식 변경이 서비스 로직에 영향을 줄 수 있다.
  - 컨트롤러와 서비스 사이의 책임이 모호해진다.
  - 쓰기 요청에 필요한 입력값과 서비스 내부 처리 기준이 섞일 수 있다.

### 2. Entity를 직접 요청/응답에 사용
- 장점
  - DTO 변환 코드가 줄어든다.
  - 초기 구현 속도가 빠르다.

- 단점
  - 내부 DB 구조가 외부 API에 노출된다.
  - 엔티티 변경이 API 변경으로 이어질 수 있다.
  - 민감한 필드가 응답에 노출될 위험이 있다.
  - 읽기 응답이 화면 요구사항이 아니라 엔티티 구조에 끌려가게 된다.

### 3. request-command-response로 분리 (선택)
- 장점
  - 외부 요청 형식과 서비스 입력 모델을 분리할 수 있다.
  - 쓰기 흐름에서는 command를 기준으로 상태 변경 의도를 명확히 표현할 수 있다.
  - 읽기 흐름에서는 response를 통해 클라이언트에 필요한 값만 반환할 수 있다.
  - 컨트롤러와 서비스 사이의 계약이 명확해진다.
  - 엔티티 구조 변경이 외부 API 응답에 직접 전파되는 것을 줄일 수 있다.
  - 계층별 책임이 분명해져 테스트와 유지보수에 유리하다.

- 단점
  - DTO 클래스 수가 늘어난다.
  - request에서 command로 변환하는 코드가 필요하다.
  - 단순 기능에서는 구조가 다소 무겁게 느껴질 수 있다.

## Consequences
- 좋은 점
  - 컨트롤러와 서비스의 책임이 명확해졌다.
  - 서비스 계층이 HTTP 요청 DTO에 직접 의존하지 않게 됐다.
  - 쓰기 요청은 command로, 읽기 응답은 response로 분리해 데이터의 사용 목적을 명확히 할 수 있었다.
  - 내부 엔티티 구조를 외부 응답과 분리할 수 있었다.
  - API 변경이 서비스 로직에 미치는 영향을 줄일 수 있었다.

- 나쁜 점 (트레이드오프)
  - DTO 파일 수가 증가했다.
  - request에서 command로 변환하는 코드가 필요해졌다.
  - 단순 CRUD에 비해 구조가 복잡해 보일 수 있다.
  - DTO 간 역할 기준이 흐려지면 오히려 중복 모델이 늘어날 수 있으므로 네이밍과 사용 위치를 일관되게 관리해야 한다.

## Evidence (선택)
- request DTO: HTTP 요청 입력 전용
- command DTO: 서비스 계층 입력 전용
- response DTO: HTTP 응답 출력 전용
- 쓰기 흐름과 읽기 흐름의 데이터 목적을 분리하기 위한 구조로 적용
